## 概述

TCP（Transmission Control Protocol）是互联网核心协议之一，提供可靠的面向连接的传输服务。为了在复杂多变的网络环境中高效传输数据，TCP 引入了流量控制和拥塞控制两大机制。本文将深入解析这两个机制的工作原理。

## 流量控制

流量控制是一种端到端的控制机制，用于防止发送方速度过快导致接收方缓冲区溢出。

### 滑动窗口协议

TCP 使用滑动窗口（Sliding Window）机制实现流量控制：

```
┌─────────────────────────────────────────────────────────┐
│                    滑动窗口示意图                         │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  已确认    │  已发送未确认  │  可发送  │  超出窗口      │
│  ─────────┼───────────────┼──────────┼────────────      │
│  │ ████ │ │ ░░░░░░░░ │ │          │                 │
│  │      │ │           │ │          │                 │
│  └──────┘ └───────────┘ └──────────┘                 │
│                                                         │
│  发送窗口 = 接收方指定的 win 大小                        │
│  实际可发送字节 = 发送 win - 已发送未确认字节数            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 工作原理

1. **窗口大小协商**：连接建立时，接收方在 SYN+ACK 报文中指定初始窗口大小（win）
2. **动态调整**：每次 ACK 报文都会携带最新的 win 值，窗口大小可变
3. **发送计算**：发送方可立即发送的字节数 = 接收方 win - 已发送但未收到 ACK 的字节数

### 零窗口问题

当接收方缓冲区满时：

1. 接收端返回 `win = 0`，发送端停止发送数据
2. 发送端启动探测定时器，定期探测窗口是否打开
3. 接收端缓冲区有空闲时，发送带有更新窗口值的 ACK 通知发送端

> **注意**：零窗口会导致传输暂停，因此在实际应用中需要谨慎设置缓冲区大小。

### 延迟确认

接收方收到数据时，会启动延迟确认定时器：

- 在定时器周期内，如果收到新的数据包，ACK 会被合并
- 定时器溢出时，发送 ACK
- 这种机制可以减少 ACK 报文数量，提高网络利用率

## 拥塞控制

拥塞控制是网络层面的控制机制，防止过多数据注入网络导致网络性能下降。

### 核心概念

| 概念 | 说明 |
|------|------|
| 拥塞窗口 (cwnd) | 发送方维护的限制 |
| 慢启动门限 (ssthresh) | 决定使用慢启动还是拥塞避免 |
| 有效窗口 | min(cwnd, 接收方 win) |

### 1. 慢启动 (Slow Start)

为什么需要慢启动？如果直接发送大量数据，当网络路由器缓存满了之后，会导致数据包丢失和重传。

**工作流程**：

```
cwnd 初始值 = 1 包
每收到一个 ACK：cwnd = cwnd + 1
每收到两个 ACK：cwnd = cwnd + 2
...
最终 cwnd = min(cwnd, 接收方 win)
```

指数增长阶段：
- 1 个 ACK → cwnd = 2
- 2 个 ACK → cwnd = 4
- 4 个 ACK → cwnd = 8

> **特点**：慢启动阶段 cwnd 呈指数增长，快速探测网络可用带宽。

### 2. 拥塞避免 (Congestion Avoidance)

当 cwnd 达到 ssthresh 时，进入拥塞避免阶段：

**算法流程**：

```
1. 初始：cwnd = 1 包，ssthresh = 65535 字节
2. 拥塞发生时：
   - ssthresh = min(cwnd, 接收方 win) / 2
   - 如果是超时导致：cwnd = 1（重新慢启动）
3. 后续判断：
   - 如果 ssthresh > cwnd：执行慢启动
   - 如果 ssthresh <= cwnd：执行拥塞避免
     每收到一个 ACK：cwnd += 1/cwnd（线性增长）
```

> **关键点**：拥塞避免阶段 cwnd 呈线性增长，谨慎地探测网络剩余容量。

### 拥塞发生场景

| 触发条件 | 处理方式 |
|---------|---------|
| 超时 | ssthresh = cwnd/2，cwnd = 1（重新慢启动） |
| 收到重复 ACK | 可能触发快重传（见下文） |

## 快重传与快恢复

当网络发生轻微拥塞时（丢包但未完全堵塞），使用快重传和快恢复机制可以避免完全重置连接。

### 工作流程

```
┌─────────────────────────────────────────────────────────┐
│                  快重传与快恢复流程                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1. 收到 3 个重复 ACK                                  │
│     ↓                                                   │
│  2. 立即重传丢失的包                                    │
│     ssthresh = cwnd / 2                                │
│     cwnd = ssthresh + 3 * MSS                         │
│     ↓                                                   │
│  3. 收到重复 ACK 时                                    │
│     cwnd += MSS，发送一个新分组                         │
│     ↓                                                   │
│  4. 收到新的 ACK（确认恢复）                            │
│     cwnd = ssthresh，进入拥塞避免算法                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 详细步骤

1. **收到 3 个重复 ACK**：
   - 立即重传丢失的数据包
   - `ssthresh = cwnd / 2`
   - `cwnd = ssthresh + 3 * MSS`

2. **收到重复 ACK**：
   - `cwnd += MSS`
   - 发送一个分组

3. **收到新的 ACK**：
   - `cwnd = ssthresh`
   - 启动拥塞避免算法

### 重复 ACK 的原因

当接收方收到一个失序的包时，会立即发送一个重复 ACK，告诉发送端需要的是哪个包（期望的序列号）。

```
发送: Seq=100, 200, 300, 400
接收: 100, 300 (丢失200)
返回: ACK=100 (期望200)
      ACK=100 (重复)
      ACK=100 (重复)
      ACK=100 (重复)  <-- 触发快重传
```

## 拥塞控制状态机

```
                    ┌──────────────┐
                    │   慢启动     │ cwnd = 1
                    │  (SS)       │ cwnd += 1/MSS
                    └──────┬───────┘
                           │ cwnd >= ssthresh
                           ▼
                    ┌──────────────┐
                    │  拥塞避免     │ cwnd += 1/cwnd
                    │  (CA)       │
                    └──────┬───────┘
                           │ 超时
                    ┌──────┴───────┐
                    │              │
                    ▼              ▼
              cwnd = 1       ssthresh = cwnd/2
              ssthresh = cwnd/2    cwnd = ssthresh
                    │              │
                    └──────┬───────┘
                           │
                    回到慢启动或拥塞避免
```

## 总结

| 机制 | 作用 | 关键参数 |
|------|------|---------|
| 流量控制 | 防止接收方缓冲区溢出 | 接收方 win |
| 慢启动 | 探测网络初始带宽 | cwnd = 1 |
| 拥塞避免 | 谨慎探测网络容量 | cwnd 线性增长 |
| 快重传 | 快速恢复丢包 | 3 个重复 ACK |
| 快恢复 | 避免完全重置连接 | cwnd 调整 |

理解 TCP 的流量控制和拥塞控制机制，对于网络性能调优、故障排查以及设计高性能网络应用都有重要参考价值。
