## 什么是物理架构设计

物理架构设计关注的是程序及其相关系统**如何部署到物理机器上**，以及如何通过机器与网络的编排来保证可靠性、可伸缩性等质量要求。

简单来说：

```
物理架构 = 硬件分布 + 软件部署 + 方案优化
```

与逻辑架构关注"怎么拆"不同，物理架构关注的是**静态位置问题**——每个软件组件跑在哪台机器上，机器之间怎么连接，硬件和软件之间如何相互影响。

---

## 核心设计目标

物理架构设计围绕两个核心问题展开：

| 核心问题 | 具体关注点 |
|----------|-----------|
| **降低开销** | 减少物理节点内的计算开销、减少节点间的通信开销 |
| **避免争用** | 避免节点内 CPU/内存/磁盘的资源争用、避免节点间网络带宽冲突 |

---

## 1. 硬件分布

硬件分布回答三个关键问题：

### 1.1 选择什么样的机器

| 维度 | 考量因素 |
|------|----------|
| **计算型** | CPU 核数、主频，适合计算密集型业务 |
| **内存型** | 大内存配置，适合缓存、内存数据库 |
| **存储型** | 大容量 SSD/HDD，适合数据库、对象存储 |
| **GPU 型** | 配备 GPU 卡，适合 AI 推理、视频转码 |
| **通用型** | 均衡配置，适合 Web 应用、微服务 |

### 1.2 物理拓扑怎么设计

机器之间如何连接，决定了通信效率和故障域：

```
                    ┌─────────┐
                    │  互联网   │
                    └────┬────┘
                         │
                  ┌──────┴──────┐
                  │   负载均衡    │
                  └──────┬──────┘
            ┌────────────┼────────────┐
       ┌────┴────┐  ┌────┴────┐  ┌────┴────┐
       │  机柜 A  │  │  机柜 B  │  │  机柜 C  │
       │ (机房 1) │  │ (机房 1) │  │ (机房 2) │
       └────┬────┘  └────┬────┘  └────┬────┘
            │            │            │
     ┌──────┴──────┐     │     ┌──────┴──────┐
     │ App + Cache │     │     │ App + Cache │
     │   DB 主     │     │     │   DB 从     │
     └─────────────┘     │     └─────────────┘
                    ┌────┴────┐
                    │ MQ 集群  │
                    └─────────┘
```

### 1.3 网络与冗余

| 设计要素 | 说明 |
|----------|------|
| **网络规范** | 万兆内网、千兆外网，VPC 隔离 |
| **总线规范** | NVMe SSD、PCIe 通道数 |
| **冗余设计** | 双网卡绑定、双电源、跨机柜部署 |
| **跨机房** | 同城双活、异地灾备 |

> **核心原则**：任何单一硬件故障都不应导致整个系统不可用。

## 2. 软件部署

软件部署回答：**每个目标单元部署在哪里、部署几个实例、以什么方式运行**。

### 2.1 部署单元规划

| 组件类型 | 部署考量 | 典型实例数 |
|----------|----------|:----------:|
| Web 前端 | 无状态，水平扩展 | 2+ |
| 应用服务 | 无状态，按负载扩展 | 3+ |
| 数据库 | 有状态，主从/分片 | 2-3（主从） |
| 缓存 | 集群模式，分片 | 3+（Redis Cluster） |
| 消息队列 | 集群模式，多副本 | 3+（Kafka Broker） |
| 搜索引擎 | 集群模式，分片+副本 | 3+（ES 节点） |
| 文件/对象存储 | 分布式存储 | 3+（多副本） |

### 2.2 系统软件选型

| 层级 | 选型 |
|------|------|
| 操作系统 | CentOS / Ubuntu / Debian |
| 容器平台 | Docker + Kubernetes |
| Web 服务器 | Nginx / Caddy |
| 应用运行时 | JVM / Go / Node.js |
| 数据库 | MySQL / PostgreSQL / MongoDB |
| 缓存 | Redis / Memcached |
| 消息队列 | Kafka / RabbitMQ / RocketMQ |

### 2.3 部署模式

| 模式 | 说明 | 适用场景 |
|------|------|----------|
| **单机部署** | 所有组件在一台机器 | 开发/测试环境 |
| **垂直拆分** | 不同组件部署在不同机器 | 小型生产环境 |
| **水平扩展** | 同一组件多实例部署 | 中大型生产环境 |
| **容器编排** | Kubernetes 管理所有服务 | 云原生架构 |

## 3. 方案优化

物理架构设计需要在多个优化目标之间取得平衡：

### 3.1 优化维度

| 维度 | 目标 | 手段 |
|------|------|------|
| **性能** | 低延迟、高吞吐 | 就近部署、SSD、万兆网络 |
| **可用性** | 故障不中断服务 | 冗余部署、自动切换、多机房 |
| **可伸缩** | 按需扩缩容 | 无状态设计、容器化、自动伸缩 |
| **经济性** | 控制成本 | 按需购买、混合云、弹性实例 |
| **技术可行性** | 团队能落地 | 选择成熟方案，避免过度设计 |
| **易维护性** | 运维友好 | 标准化部署、监控完善、文档齐全 |

### 3.2 四个关键优化问题

**问题 1：如何降低节点内的计算开销**

- CPU 密集型和 I/O 密集型任务分开部署
- 合理设置进程/线程数（与 CPU 核数匹配）
- 使用连接池、对象池减少创建销毁开销
- 本地缓存减少重复计算

**问题 2：如何降低节点间的通信开销**

- 高频交互的服务部署在同一机柜/机房
- 使用内网通信，避免走公网
- 批量传输替代逐条请求
- 异步消息替代同步调用

**问题 3：如何避免节点内资源争用**

- 数据库和应用服务器分开部署（避免 CPU/内存争用）
- I/O 密集型服务使用独立磁盘或 SSD
- 设置资源限额（cgroups、Kubernetes resource limits）
- 大查询和在线服务隔离（读写分离）

**问题 4：如何避免节点间网络带宽冲突**

- 核心链路使用万兆网络
- 备份/同步流量走独立网络通道
- CDN 分流静态资源
- 限流和流量整形，防止突发流量打满带宽

## 设计检查清单

在完成物理架构设计后，用以下清单自检：

- [ ] 每个关键组件是否有冗余（无单点故障）
- [ ] 数据库是否有主从/备份方案
- [ ] 高频交互的服务是否就近部署
- [ ] 是否考虑了跨机房/跨可用区容灾
- [ ] 网络带宽是否满足峰值流量需求
- [ ] 是否有监控和告警覆盖所有节点
- [ ] 扩容方案是否明确（加机器即可扩展）
- [ ] 成本是否在预算范围内
- [ ] 运维团队是否有能力管理这套架构
