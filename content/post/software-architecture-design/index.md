## 什么是架构设计

架构设计用来描述系统的整体组成——包含多少子系统、模块，以及它们之间的关系。好的架构设计不是追求技术复杂度，而是在**功能需求**和**质量需求**之间找到平衡。

### 架构设计的两大驱动力

| 驱动力 | 关注点 |
|--------|--------|
| **功能需求** | 系统要做什么——业务功能、用户场景 |
| **质量需求** | 系统要做到什么程度——高可用、高并发、安全性、可伸缩、可扩展、可维护 |

## 架构设计六大维度

### 1. 逻辑架构

逻辑架构回答的是"系统怎么拆"——如何将一个大系统划分为多个子系统或模块。

**划分原则**：

| 原则 | 说明 | 示例 |
|------|------|------|
| **按参与者划分** | 不同角色使用的功能归入不同子系统 | 买家系统 vs 卖家系统 vs 管理后台 |
| **按产品分类** | 不同类型的产品/业务独立系统 | 支付系统 vs 商品系统 vs 订单系统 |
| **兼顾规模** | 功能简单的系统不必拆得过细 | 小型内部工具无需微服务化 |
| **按质量需求** | 压力大的模块单独拆分 | 秒杀服务从主站独立出来 |

> **核心思想**：高内聚、低耦合。每个子系统聚焦于一个明确的业务域。

### 2. 总体架构

绘制总体架构图，展示各子系统及它们之间的关系：

```
┌─────────────────────────────────────────────────┐
│                    接入层                        │
│          负载均衡 / API Gateway / CDN            │
├─────────────────────────────────────────────────┤
│                    应用层                        │
│    用户服务  │  订单服务  │  支付服务  │  ...     │
├─────────────────────────────────────────────────┤
│                    服务层                        │
│    消息队列  │  缓存服务  │  搜索引擎  │  ...     │
├─────────────────────────────────────────────────┤
│                    数据层                        │
│       关系数据库  │  NoSQL  │  对象存储           │
└─────────────────────────────────────────────────┘
```

### 3. 集成架构

描述本系统与**外部依赖系统**之间的关系和接口：

- 上游系统：谁调用我？通过什么协议？
- 下游系统：我依赖谁？接口契约是什么？
- 第三方服务：支付网关、短信服务、地图 API 等

> **关键问题**：每个外部依赖都是潜在的故障点，需要设计降级、熔断和超时策略。

### 4. 开发架构

核心是**技术选型**，选择合适的框架和工具：

| 选型原则 | 说明 |
|----------|------|
| **可控制** | 团队能在短时间内了解并精通使用 |
| **可扩展** | 在特定场景下易于扩展，优先选择开源、轻量级方案 |
| **高可用** | 框架本身质量可靠，社区活跃 |
| **高效率** | 框架处理性能高，能减少重复工作，易于调优 |
| **低成本** | 采购成本和学习成本都在可接受范围内 |
| **可维护** | 文档齐全，最好有商业支持或活跃社区 |

> **常见误区**：盲目追新技术。选型应以团队能力和业务需求为导向，而非技术热度。

### 5. 数据架构

关注数据如何建模和持久化：

- **数据模型**：ER 图设计、领域模型、数据字典
- **持久化方案**：关系型数据库、NoSQL、文件存储、对象存储的选择和搭配
- **数据流转**：数据从产生到消费的完整链路

### 6. 部署架构

描述系统在物理或云环境中的部署拓扑：

- 数据库集群（主从、分片）
- 分布式缓存（Redis 集群）
- 消息中间件（Kafka、RabbitMQ）
- 应用服务器（容器编排、自动扩缩容）
- 网络拓扑（VPC、安全组、负载均衡）

## 关键质量需求设计

### 安全性

| 威胁 | 防护措施 |
|------|----------|
| **篡改** | 数字签名、HTTPS、数据完整性校验 |
| **抵赖** | 操作日志、审计追踪 |
| **窃取** | 数据加密（传输加密 + 存储加密）、最小权限原则 |
| **攻击** | WAF、限流、输入校验、SQL 注入/XSS 防护 |

### 高可用

确保系统在部分故障时仍能正常提供服务：

| 策略 | 说明 |
|------|------|
| **消除单点** | 每个关键组件至少两个实例 |
| **集群化** | 双节点或多节点集群，自动故障切换 |
| **监控报警** | 实时监控 + 异常自动报警 |
| **自动恢复** | 健康检查 + 自动重启/替换故障实例 |
| **流量控制** | 限流、熔断、降级，防止雪崩 |
| **故障隔离** | 应用拆分、舱壁模式、灰度发布 |

### 高性能

```
请求 → CDN（静态资源）→ 负载均衡 → 应用集群 → 缓存 → 数据库
         ↑                              ↑
     尽早返回                       尽量命中缓存
```

| 层级 | 优化手段 |
|------|----------|
| **静态资源** | CDN 加速、图片/视频分离、浏览器缓存 |
| **缓存** | 应用缓存、Web 缓存、分布式缓存（Redis） |
| **异步处理** | 消息队列削峰填谷，耗时操作异步化 |
| **架构伸缩** | 无状态设计，支持水平扩展 |
| **负载均衡** | 多级负载均衡（DNS → LB → 应用） |
| **数据库** | 分库分表分区、索引优化、读写分离、SQL 调优 |
| **应用层** | 消除不合理逻辑、消除重复操作、连接池复用 |

### 可伸缩

通过增加硬件来提升系统吞吐量，要求架构设计时就考虑：

| 维度 | 策略 |
|------|------|
| **应用层** | 垂直拆分（按业务域拆服务），无状态设计 |
| **数据层** | 水平拆分（分表分库），数据分片 |
| **计算层** | 容器化部署，自动扩缩容（HPA） |

## 架构设计的输出物

一份完整的架构设计文档通常包含：

1. **需求概述**：关键功能需求 + 质量需求
2. **逻辑架构图**：子系统划分及职责
3. **总体架构图**：分层视图和组件关系
4. **集成架构图**：外部系统依赖和接口
5. **技术选型表**：框架、中间件、数据库选择及理由
6. **数据模型**：核心 ER 图和数据字典
7. **部署架构图**：物理/云部署拓扑
8. **质量保障方案**：安全、可用、性能、伸缩的具体措施
