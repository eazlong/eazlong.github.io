## 概述

TCP（Transmission Control Protocol）是一种面向连接的可靠传输协议，为了保证数据的可靠传输，TCP 内部维护了多个定时器。这些定时器在连接管理、数据传输、错误恢复等各个环节发挥着关键作用。本文将详细介绍 TCP 的四种核心定时器。

## TCP 定时器一览

TCP 协议主要使用以下四种定时器：

| 定时器 | 英文名称 | 主要作用 |
|--------|----------|----------|
| 重传定时器 | Retransmission Timer | 可靠传输，检测丢包并重传 |
| 坚持定时器 | Persist Timer | 解决零窗口死锁 |
| 保活定时器 | Keepalive Timer | 检测连接是否存活 |
| 2MSL 定时器 | 2MSL Timer | 等待连接完全关闭 |

## 1. 重传定时器 (RTO Timer)

重传定时器是 TCP 中最重要的定时器，用于确保数据的可靠传输。

### 工作原理

```
┌─────────────────────────────────────────────────────────┐
│                  重传定时器工作流程                       │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  发送数据                                               │
│     ↓                                                   │
│  启动重传定时器                                         │
│     ↓                                                   │
│  ┌─────────────────────┐                                │
│  │   等待 ACK          │                                │
│  │   (定时器运行中)    │                                │
│  └─────────┬───────────┘                                │
│            │                                            │
│     ┌──────┴──────┐                                     │
│     ↓             ↓                                     │
│  收到 ACK      超时                                      │
│     ↓             ↓                                     │
│  取消定时器     重传数据                                 │
│  继续发送      重新启动定时器（指数退避）                   │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

### 关键特性

- **超时判定**：发送数据后启动，如果在规定时间内未收到 ACK，则判定为丢包
- **指数退避**：每次重传超时，时间间隔翻倍（首次 1s，第二次 2s，4s...）
- **RTT 测量**：TCP 会动态测量往返时间，调整超时值（RTO）

### 超时重传 vs 快速重传

| 方式 | 触发条件 | 优点 |
|------|---------|------|
| 超时重传 | 定时器超时 | 简单可靠 |
| 快速重传 | 收到 3 个重复 ACK | 更快的丢包恢复 |

> **最佳实践**：现代 TCP 实现通常优先使用快速重传，因为它比重传定时器超时更快地检测到丢包。

## 2. 坚持定时器 (Persist Timer)

坚持定时器用于解决"零窗口"导致的死锁问题。

### 问题背景

当接收方缓冲区满时，会返回 `win = 0`（窗口大小为 0），发送方停止发送数据。但当接收方缓冲区有空闲时，会发送带有更新窗口值的 ACK。如果这个 ACK 丢失，发送方将永远等待，连接陷入死锁。

### 工作原理

```
发送方收到 win=0
     ↓
停止发送数据
     ↓
启动坚持定时器（通常 60s）
     ↓
┌─────────────────────┐
│ 定时器到期          │
│ 发送窗口探测报文    │  ← 探针只是 1 字节，不含数据
└─────────┬───────────┘
          ↓
   ┌──────┴──────┐
   ↓             ↓
 收到 ACK   未收到 ACK
   ↓             ↓
 根据 win  重新启动坚持定时器
 恢复发送
```

### 特点

- **探测间隔**：通常为 60 秒
- **探针内容**：仅 1 字节的探测报文，不含实际数据
- **持续探测**：直到窗口打开或连接关闭

> **注意**：坚持定时器确保了即使窗口更新通知丢失，连接也不会永久阻塞。

## 3. 保活定时器 (Keepalive Timer)

保活定时器用于检测连接的对端是否仍然存活。

### 工作原理

```
连接空闲（双方无数据交换）
     ↓
启动保活定时器（通常 2 小时）
     ↓
┌─────────────────────┐
│ 定时器到期          │
│ 发送保活探测报文    │
└─────────┬───────────┘
          ↓
   ┌──────┴──────┐
   ↓             ↓
 收到响应    未收到响应
   ↓             ↓
 重置定时器  继续探测（通常 10 次）
   ↓             ↓
 连接正常    判定连接失效，关闭连接
```

### 配置参数（Linux 系统）

```bash
# 查看当前配置
cat /proc/sys/net/ipv4/tcp_keepalive_time   # 首次探测时间（默认 7200s = 2h）
cat /proc/sys/net/ipv4/tcp_keepalive_intvl  # 探测间隔（默认 75s）
cat /proc/sys/net/ipv4/tcp_keepalive_probes # 探测次数（默认 9 次）
```

### 应用场景

| 场景 | 说明 |
|------|------|
| 检测客户端崩溃 | 客户端异常断电、网络中断 |
| 检测连接泄露 | 防止半开连接占用资源 |
| 心跳检测 | 某些应用需要感知连接状态 |

> **注意**：保活定时器默认关闭，如需使用可通过 `setsockopt` 启用。对于需要长时间保持连接的应用（如 WebSocket），通常在应用层实现心跳机制。

## 4. 2MSL 定时器 (2MSL Timer)

2MSL 定时器用于确保连接完全关闭，以及处理延迟的报文段。

### MSL 是什么

MSL（Maximum Segment Lifetime）是报文段在网络中的最大生存时间，TCP 实现通常设为 2 分钟（120 秒）。

### 工作原理

```
主动关闭方发送 FIN，进入 FIN_WAIT_1
     ↓
收到被动方 ACK，进入 FIN_WAIT_2
     ↓
收到被动方 FIN，进入 TIME_WAIT
     ↓
启动 2MSL 定时器（通常 60-120 秒）
     ↓
┌─────────────────────┐
│ 定时器到期          │
│ 确保延迟报文被处理  │
│ 确保后续连接不受影响 │
└─────────┬───────────┘
          ↓
     关闭连接，进入 CLOSED
```

### 2MSL 的作用

1. **处理延迟的 ACK/FIN**：确保在连接关闭前，所有在网络中延迟的报文都被处理
2. **避免旧连接数据**：防止相同端口的新连接收到旧连接的延迟数据
3. **可靠关闭**：确保被动关闭方收到最后的 ACK

### 常见问题

#### TIME_WAIT 状态堆积

在高并发服务器上，大量短连接可能导致 TIME_WAIT 状态堆积：

| 解决方案 | 说明 |
|---------|------|
| 开启端口复用 | `tcp_tw_reuse = 1` |
| 减少 MSL 时间 | `tcp_fin_timeout = 30` |
| 使用长连接 | 减少频繁的连接建立/关闭 |

> **最佳实践**：高并发服务应合理配置 TIME_WAIT 参数，避免端口耗尽。

## 总结

| 定时器 | 触发条件 | 超时后果 | 典型值 |
|--------|---------|---------|--------|
| 重传定时器 | 发送数据未收到 ACK | 重传数据（指数退避） | 动态计算 |
| 坚持定时器 | 收到 win=0 | 发送窗口探测报文 | 60s |
| 保活定时器 | 连接空闲 | 发送保活探测 | 2h 后开始 |
| 2MSL 定时器 | 主动关闭 | 连接关闭 | 2×MSL |

理解这四种定时器的工作原理，对于诊断网络问题、优化 TCP 性能以及设计可靠的网络应用都有重要帮助。
